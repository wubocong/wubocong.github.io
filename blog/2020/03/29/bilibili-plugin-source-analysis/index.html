B站下载助手Chrome插件源码分析

2020-03-29 Comments JavaScript

新装了一个Chrome插件B站下载助手，打开b站视频会出现这样画面，那么它是如何实现的呢？

打开开发者工具，点击合并下载，可以看到network里出现了一个7.8M的fetch请求，这个请求毫无疑问就是视频的下载，这个地址是从哪里来的呢？为了方便查看，我们将Initiator相同的合并查看，发现来自插件(VM1135)共有三个请求（下图），另外两个请求分别又是干什么的呢？

我们先看playurl?cid=83089367&bvid=BV1sb411t7ps&otype=json&qn=80这个请求，打开其response，我们看到其data中正好包含了视频的地址，看来这个请求就是用来获取视频下载地址的。

使用VS Code打开位于Chrome个人文件夹下的插件目录，发现源代码位于cs.js中，美化后查看其源代码，搜索fetch发现如下代码。

可以看出其请求了 api.bilibili.com，请求的参数有cid，avid|bvid，qn三个，这三个参数分别来自哪里呢？代码经过丑化变量都被重命名了，但是VS Code提供了一个Rename Symbol的功能，用来批量重命名一个变量，它会自动根据AST（抽象语法树）分析出一个单词对应的变量在哪里，即使是重名的也可以根据作用域分辨出来，使用方法就是按下F2，因此我们很快找出了三个变量所在的位置（如上图箭头所示）。

这三个变量又是怎么获得的呢，这涉及到上图中的另一个请求，它直接fetch了视频所在页面的html，我们查看页面源码可以发现script标签中定义了一个全局变量INITIAL_STATE（如下图），这是一个json对象，包含许多关键信息。上述另一个fetch请求获取到html后通过正则获取到它并parse为json。

cs.js中还有两处使用了fetch，其中一处如下图，这一处是用来获取番剧的视频地址的，而前面的是获取普通视频的下载地址，原理相同，不展开。

视频获取后插件会自动进行下载，这就是最后一处fetch的作用，我将其变量命名后改写为如下：

这里运用了ReadableStream，是fetch API的新特性，具体可查看Using readable streams，下载完成后将用blob对象生成blob协议的URL赋值给a标签的href属性，然后自动dispatch一个click事件完成下载。

如此其原理就基本明白了，通过请求api.bilibili.com获取视频flv地址，并通过ReadableStream下载。不过如果将拿到的视频地址直接放进下载器是下载不了的，因为b站会检验http请求的Referer，得构造一个带Referer的fetch请求才行。
